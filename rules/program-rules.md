# Program-Specific Rules

Dos and don'ts for each grant program. Load alongside `grant-rules.md` when helping with a specific program.

---

## Solana Foundation

**Always do:**
- Frame the project as a public good — who in the ecosystem benefits beyond the founder
- Ask for GitHub links before writing anything — Foundation reviewers check these
- Include a competitive landscape — name similar projects and differentiate specifically
- Structure milestones with verifiable links (GitHub release, live URL, install count)
- Mention open-source license explicitly in the application
- Reference any prior grants delivered — Foundation values proven execution

**Never do:**
- Recommend Foundation to a founder with no Solana GitHub contributions
- Write "large market opportunity" without a specific number
- Allow a budget line that just says "development" with no breakdown
- Submit without a clear public goods case — commercial projects need the convertible grant path
- Promise milestones longer than 6 weeks without a strong justification
- Write the application before checking if a similar project was already funded

**Red flags to flag immediately:**
- No GitHub link in the team section
- Final milestone is "launch" with no definition of what launch means
- Budget over $50K for a first-time Foundation applicant with no track record
- Problem statement that could describe any blockchain, not specifically Solana

## Superteam Earn

**Always do:**
- Keep the application under 600 words — Superteam reviewers read fast
- Lead with regional angle — why does this matter specifically for your region
- Include Superteam Earn profile link in the team section
- Reference any prior bounty wins — even small ones count
- Make M1 deliverable within 2-3 weeks — shows momentum
- Name the regional chapter lead if known — shows ecosystem familiarity

**Never do:**
- Submit without at least 2 completed Superteam Earn submissions on profile
- Write a Foundation-style long-form application for Superteam
- Propose something that only benefits the founder, not the community
- Use corporate language — Superteam tone is builder-to-builder
- Apply to a regional grant without a clear regional angle
- Ask for more than $10,000 as a first-time Superteam grant applicant

**Red flags to flag immediately:**
- Empty Superteam Earn profile (go win bounties first)
- Application over 1,000 words
- No regional community angle
- Milestones with no timeline (just phases, no weeks)
- Budget with no line items

## Colosseum

**Always do:**
- Ask if they have a working demo before anything else — no demo = not ready
- Emphasize the demo video — judges watch it before reading anything
- Frame the project as a startup, not a tool — "could this be a company?"
- Keep the GitHub README clean with a working install path
- Include a market thesis — who pays for this eventually?
- Mention team composition — Colosseum favors teams over solo founders

**Never do:**
- Submit to Colosseum with no working demo
- Use AI voiceover in the demo video
- Build a slide deck as the primary submission — code first, slides second
- Pick the most crowded track without a strong differentiation (DeFi is hardest)
- Submit a solo project without addressing how you'll scale the team
- Copy the structure of last year's winning projects — judges notice patterns

**Red flags to flag immediately:**
- Demo video is a slide deck, not a working product
- No GitHub repo or empty repo
- "We will build X" with no current progress
- Team of 1 with no plan for adding technical co-founder
- Market section says "the market is huge" with no specifics

## Cross-program rules

**When helping with multiple programs simultaneously:**
- Always write separate applications — never submit the same text to both
- Superteam frame = regional impact, personal story, community benefit
- Foundation frame = ecosystem-wide public good, technical depth, open source
- Colosseum frame = startup potential, product impact, team that can scale
- Budget amounts should differ — Foundation can go higher than Superteam
- Timeline can overlap — applying simultaneously is encouraged, not discouraged
