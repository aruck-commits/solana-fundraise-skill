# Writing Rules

Specific rules for grant application language, tone, and structure. Apply whenever writing or reviewing any grant application copy.

---

## Language rules

**Every claim needs proof**
- Number, link, or name — no exceptions
- "Many developers struggle" → "47 developers surveyed across 3 regions"
- "Large market" → "2,300 new tokens launched on Solana last month"
- "Experienced team" → "github.com/username — 200+ commits to Solana repos in 6 months"

**Lead with pain, not solution**
- The problem section must make the reviewer feel the pain before they know what you're building
- If the first sentence mentions the product name, rewrite it
- If the first paragraph could be removed without losing information — remove it

**One idea per sentence**
- Long compound sentences with multiple clauses hide weak thinking
- If a sentence has more than one "and" — split it
- If a sentence is over 25 words — split it

**Active voice only**
- "We will build" not "the tool will be built"
- "Developers lose 3 hours" not "3 hours are lost by developers"
- "I shipped SolProbe" not "SolProbe was shipped"

---

## Words and phrases to never use

These appear in almost every weak grant application. Flag and replace immediately:

| Banned phrase | Replace with |
|---|---|
| "innovative" | describe specifically what's new |
| "cutting-edge" | describe the specific technology |
| "passionate about" | show the work, skip the adjective |
| "leverage" (as a verb) | "use" |
| "ecosystem" (as filler) | only use when referring to specific projects/people |
| "seamless" | describe the actual user experience |
| "scalable" | describe how it scales specifically |
| "robust" | describe what makes it reliable |
| "large market opportunity" | give the actual number |
| "we believe" | state it as fact or provide evidence |
| "best in class" | compared to what, specifically? |
| "comprehensive solution" | describe what it actually does |
| "end to end" | describe the specific start and end points |
| "world class" | never |
| "passionate team" | link to their GitHub |

## Structure rules

**Problem section**
- Must name a specific user (not "developers" — "Solana developers deploying their first program")
- Must include a number (affected users, time lost, money lost, projects abandoned)
- Must explain why the problem exists (root cause)
- Must be written so a stranger understands the pain without knowing your solution
- Maximum 150 words for Superteam, 250 words for Foundation

**Solution section**
- First sentence names the product and the core action ("SolProbe detects and explains Solana validator errors in real time")
- Second sentence explains how it works technically — one sentence, no buzzwords
- Third section: 3 specific things a user can do after this exists that they can't do today
- No marketing language — describe behavior, not benefits

**Milestones**
- Table format always — never prose milestones
- Every deliverable is a URL, a GitHub link, a number, or a published document
- First milestone within 3 weeks always
- Last milestone is always something live and public
- Milestone names are nouns, not verbs ("Public alpha" not "Ship the alpha")

**Team section**
- Links before descriptions — GitHub first, then Superteam profile, then X
- Achievements before roles — "shipped X with Y users" before "senior developer"
- If solo: address it directly — don't hide it, explain how you'll cover the gaps
- Maximum 100 words per team member

**Budget**
- Line items always — never a lump sum
- Developer time = hours or weeks × rate
- Rate should match location (Mumbai rate ≠ San Francisco rate — be honest)
- Every line item should map to a specific milestone
- Round numbers over $1,000 get scrutinized — justify them

---

## Review checklist — run before every submission

**Language:**
- [ ] Every claim has a number, name, or link
- [ ] No banned phrases from the list above
- [ ] All sentences under 25 words
- [ ] Active voice throughout
- [ ] Problem section doesn't mention the product name

**Structure:**
- [ ] Problem section names a specific user and includes a number
- [ ] Solution section has a clear one-liner in the first sentence
- [ ] Milestones are in table format with verifiable deliverables
- [ ] First milestone is within 3 weeks
- [ ] Final milestone is something live and public
- [ ] Team section has links before descriptions
- [ ] Budget is broken into line items

**Program fit:**
- [ ] Application length matches the program (Superteam ≤600 words, Foundation ≤1,500)
- [ ] Frame matches the program (regional for Superteam, ecosystem for Foundation)
- [ ] Competitive landscape addressed (name similar projects and differentiate)

**Final check:**
- [ ] Read it out loud — anything that sounds like a press release needs to be rewritten
- [ ] Someone else has read it and flagged anything vague
- [ ] All links are live and go to the right place
