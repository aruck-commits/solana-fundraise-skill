# Rejection Handling

Getting rejected is part of the process. Every serious Solana founder has been rejected at least once. What separates funded founders from unfunded ones isn't the rejection — it's what happens after.

---

## First: why applications get rejected

Most rejections fall into one of six categories:

**1. Wrong program for your stage**
Applied to Foundation with no GitHub history. Applied to Colosseum with no working demo. The program wasn't wrong — the timing was.

**2. Vague problem statement**
"Many developers struggle with X" with no number, no specific user, no proof you've seen the problem firsthand.

**3. Unverifiable milestones**
"Complete development" is not a milestone. Reviewers can't pay out what they can't verify.

**4. No proof of work**
No GitHub, no Superteam profile, no community presence. They can't tell if you can execute.

**5. Already funded**
A similar project already received a grant. You didn't mention it or differentiate.

**6. Budget mismatch**
Asking for $50K for something that looks like a $10K project, or vice versa.

---

## How to get feedback after a rejection

Most programs won't proactively tell you why. You have to ask — and how you ask matters.

**Message format:**
```
Hi [Name],

Thanks for reviewing my application for [Project Name].

I understand the decision — I'd genuinely like to improve and reapply 
when the time is right. Would you be able to share one or two specific 
reasons it didn't move forward? Even a line or two would help me 
understand what to address.

Happy to share any additional context if useful.

[Your name]
```

Keep it short. No defensiveness. No asking them to reconsider.

**Where to send it:**
- Superteam: DM the regional lead on Discord or X
- Foundation: Reply to the rejection email, or reach out to grants@solana.org
- Colosseum: Reach out via Discord or X to the team

**Response rate:** Maybe 50%. Don't be discouraged if you don't hear back. Even one line of feedback is gold.

---

## The 4-week reapplication playbook

Don't reapply immediately with the same application. Fix it first.

**Week 1: Diagnose**
- List every vague sentence in your application
- List every claim without a number or link
- List every milestone that isn't verifiable
- Ask yourself honestly: did the problem section make you feel the pain?

**Week 2: Fix the application**
- Rewrite the problem section from scratch
- Add real numbers to every claim
- Replace every vague milestone with a verifiable one
- Add any proof of work you've built since the rejection

**Week 3: Build more proof**
- Win a Superteam bounty if you don't have one
- Push code to GitHub
- Post publicly about the problem you're solving
- Get one person to publicly say they'd use your product

**Week 4: Reapply**
- New application, not a slightly edited old one
- Open with what's changed since the last application if reapplying to the same program

---

## What to say when reapplying to the same program

If you're reapplying to a program that rejected you:

```
Note: I applied to [Program] in [Month] and was not selected. 
Since then I've [specific thing: won 2 Superteam bounties / shipped a prototype / 
onboarded 50 users]. This application addresses the feedback I received 
about [specific issue].
```

Acknowledge it. Don't hide it. Reviewers respect founders who learn and come back stronger.

---

## When to try a different program instead

If you've been rejected twice by the same program, try a different one before reapplying again.

| Rejected from | Try instead |
|---|---|
| Solana Foundation | Superteam grant (build more proof first) |
| Superteam grant | Superteam bounties (build profile first) |
| Colosseum hackathon | Enter next cycle with a better demo |

The stacking playbook in `agents/grant-stacking.md` shows how to sequence programs after a rejection.

---

## The mindset that gets founders funded

The founders who eventually get funded aren't the ones with the best first application. They're the ones who treat every rejection as a diagnostic — a specific list of things to fix — and come back with a better version.

Rejection is data. Use it.
