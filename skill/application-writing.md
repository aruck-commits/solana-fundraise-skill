# Writing a Winning Grant Application

Most applications fail not because the idea is bad — but because the writing is weak. Here's how to fix that.

---

## The basics that most people get wrong

**Lead with the problem, not your solution**

Reviewers don't care about your product until they care about the problem. Make them feel the pain first.

Bad: "We're building a cross-chain liquidity aggregator on Solana."
Good: "Every Solana builder launching a new token manually manages 4 DEX pools. It takes hours, breaks constantly, and nobody has unified it."

**Be specific or be ignored**

Every claim needs a number, a name, or a link. "There is a large market for this" gets skipped. "2,300 tokens launched on Solana last month, every one hit this problem" gets read.

**Milestones are promises**

Never write a milestone you can't hit. Reviewers pay out based on milestones. One miss damages the relationship with that program permanently.

**Your team section is your credibility**

Link everything — GitHub, X, Superteam Earn profile, prior projects. No links = no credibility. If you have nothing to link yet, go win 2-3 Superteam bounties first, then apply.

**Match length to the program**
- Superteam: 400–600 words max
- Solana Foundation: 800–1,500 words
- Colosseum: clean README + 3-min video

---

## Problem section

```
[Target user] currently struggle with [specific problem].
This happens because [root cause].
The result is [concrete negative outcome — time lost, money lost, projects failing].
[X] builders/projects/users face this today.
Nobody has solved it because [honest reason].
```

Write this. Then cut any sentence that doesn't add new information.

---

## Solution section

```
[Project name] is a [type of tool] that [core action] for [target user].

It works by: [1-2 sentence technical explanation, no buzzwords]

In 6 months, a builder using [project name] will be able to:
- [specific thing 1]
- [specific thing 2]
- [specific thing 3]
```

## Milestones

| # | Milestone | Deliverable | Timeline | Amount |
|---|---|---|---|---|
| M1 | [Name] | [GitHub link / live URL / doc] | Week X | $X |
| M2 | [Name] | [GitHub link / live URL / doc] | Week X | $X |
| M3 | [Name] | [GitHub link / live URL / doc] | Week X | $X |

Rules:
- Every deliverable must be verifiable by someone who doesn't know you
- No milestone longer than 6 weeks
- First milestone within 2–4 weeks
- Final milestone = something live, not "complete development"

---

## Budget

```
Total: $X

- Development (X weeks × $Y/week): $Z
- Design: $Z
- Infrastructure / hosting: $Z
- Testing / audit: $Z
- Docs: $Z
```

Be honest. Reviewers know what things cost. Vague line items get pushback.

---

## The "why you" section — rank these by impact

1. Prior grants received and delivered
2. Superteam Earn bounty wins (with links)
3. GitHub with Solana contributions
4. Community you run (size, activity)
5. Hackathon placements
6. Relevant work experience
7. Education (weakest — only include if directly relevant)

---

## Before you submit

- [ ] Problem section lands without knowing your solution
- [ ] Every claim has a number, name, or link
- [ ] Milestones are verifiable and time-bound
- [ ] Budget is broken into line items
- [ ] Team section has links
- [ ] Right length for this program
- [ ] Someone else read it and flagged anything vague
- [ ] No sentence that sounds like it came from ChatGPT
