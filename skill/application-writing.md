# Writing a Winning Grant Application

Most grant applications fail not because the idea is bad — but because the writing is weak. This file gives you the templates, principles, and red flags to write applications that actually get funded.

---

## The universal grant writing principles

**1. Lead with the problem, not your solution**
Grant reviewers don't care about your product until they care about the problem. Spend your first paragraph making them feel the pain.

Bad: "We're building a cross-chain liquidity aggregator on Solana."
Good: "Every Solana builder who needs liquidity for a new token has to manually manage 4 different DEX pools. It takes hours, breaks constantly, and nobody has built a unified layer for it."

**2. Be specific or be ignored**
Vague = untrustworthy. Every claim needs a number, a name, or a link.

Bad: "There is a large market for this."
Good: "Over 2,300 new tokens launched on Solana last month. Each one faced this problem."

**3. Milestones are promises — treat them that way**
Never write a milestone you can't hit. Reviewers fund based on milestones and pay out based on milestones. One missed milestone kills your relationship with that program.

**4. Your team section is your credibility**
Link everything: GitHub, X, Superteam Earn profile, prior projects. If you have no links, you have no credibility. Start building your public profile before you apply.

**5. Match the length to the program**
- Superteam: 400–600 words max
- Solana Foundation: 800–1,500 words
- Colosseum hackathon: concise README + 3-min video

## Problem section template

```
[Target user] currently struggle with [specific problem].
This happens because [root cause].
The result is [concrete negative outcome — time lost, money lost, projects failing].
[X number] of builders/projects/users face this today.
Nobody has properly solved it because [honest reason].
```

Fill in the blanks. Then cut any sentence that doesn't add new information.

---

## Solution section template

```
[Project name] is a [type of tool/protocol/platform] that [core action verb] for [target user].

It works by: [1-2 sentence technical explanation — no buzzwords]

In 6 months, a builder using [project name] will be able to: [3 specific things they can do that they can't do today]
```

## Milestones template

Use this table format for every application:

| # | Milestone | Deliverable (link-able proof) | Timeline | Amount |
|---|---|---|---|---|
| M1 | [Name] | [GitHub commit / live URL / doc link] | Week [X] | $[X] |
| M2 | [Name] | [GitHub commit / live URL / doc link] | Week [X] | $[X] |
| M3 | [Name] | [GitHub commit / live URL / doc link] | Week [X] | $[X] |

Rules for good milestones:
- Every deliverable must be verifiable by someone who doesn't know you
- No milestone longer than 6 weeks
- First milestone in 2–4 weeks (shows momentum)
- Final milestone = something live and usable, not "complete development"

---

## Budget template

```
Total requested: $[X]

Breakdown:
- Development (X weeks × $Y/week): $Z
- Design (if applicable): $Z
- Infrastructure / hosting (monthly): $Z
- Testing / audit (if applicable): $Z
- Documentation: $Z
```

Don't hide developer cost inside vague line items. Reviewers know what development costs — be honest.

## The "why you" section — what to include

Rank these in order of impact (include whatever you have):

1. Prior grants received and delivered (strongest signal)
2. Superteam Earn bounty wins (links)
3. GitHub profile with Solana contributions
4. Community you run or contribute to (size, activity)
5. Hackathon placements
6. Relevant work experience
7. Education (weakest — only include if directly relevant)

If you have nothing from this list yet: win 2–3 Superteam bounties first, then apply.

---

## Before you submit — checklist

- [ ] Problem section makes someone feel the pain without knowing your solution
- [ ] Every claim has a number, name, or link
- [ ] Milestones are verifiable and time-bound
- [ ] Budget is broken down by line item
- [ ] Team section has links (GitHub, X, Superteam profile)
- [ ] Application is the right length for this program
- [ ] Someone else has read it and flagged anything that sounds vague or AI-generated
- [ ] Spell-checked and grammar-checked
